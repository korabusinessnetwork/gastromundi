**Status:** Revisado (parcialmente supersedido)  
**Data:** 2026-06-15  
**Decisores:** A preencher (founders + tech lead)  
**Supersede:** N/A  
**Supersedido por:** [ADR-002](./adr-002.md) — Supabase passa a ser apenas o PostgreSQL gerenciado; auth via Clerk; acesso a dados via API Express + Drizzle (não SDK direto no frontend).

## Contexto

GastroMundi está sendo construído do zero como um SaaS. Precisamos de uma stack que permita:

- Velocidade de desenvolvimento alta no early stage (time pequeno, MVP rápido)
- Sem necessidade de gerenciar infraestrutura de servidor própria no início
- Boa DX (Developer Experience) com tipagem forte
- Path claro de escalabilidade conforme o produto cresce
- Baixo custo operacional no início (antes de ter receita)

O produto é uma aplicação web com autenticação de usuários, banco de dados relacional, armazenamento de arquivos e potencial de funcionalidades em tempo real.

## Decisão

Adotamos a seguinte stack:

| Camada | Tecnologia | Versão |
|--------|-----------|--------|
| Frontend framework | React | 18.x |
| Build tool | Vite | 5.x |
| Estilização | Tailwind CSS | 3.x |
| Backend / BaaS | Supabase | — |
| Banco de dados | PostgreSQL (via Supabase) | 15.x |
| Linguagem | TypeScript | 5.x |
| Roteamento | React Router | 6.x |

## Alternativas Consideradas

### Next.js + Supabase
- **Prós:** SSR nativo, melhor SEO, file-based routing, grande ecossistema
- **Contras:** Maior complexidade de deploy, overhead desnecessário para app SaaS autenticado (SEO não é crítico para páginas atrás de login), maior curva de aprendizado para o time atual

### Remix + Supabase
- **Prós:** Web standards, ótimo para formulários, data loading explícito
- **Contras:** Menor adoção, ecossistema menor, curva de aprendizado mais íngreme, time menos familiarizado

### React + Vite + Firebase
- **Prós:** Bem estabelecido, muita documentação
- **Contras:** Firebase tem lock-in mais agressivo, custo pode escalar mal, PostgreSQL é superior ao Firestore para dados relacionais, migração futura seria mais custosa

### React + Vite + Backend Express próprio
- **Prós:** Controle total, sem lock-in de BaaS
- **Contras:** Overhead operacional significativo no early stage (deploy, manutenção, SSL, escalabilidade), distrai do produto principal, custo maior antes de ter receita

## Consequências

**Positivas:**
- Desenvolvimento rápido sem gerenciar infraestrutura de servidor
- PostgreSQL com RLS oferece segurança robusta por padrão
- Supabase Auth elimina implementação de auth do zero
- Vite oferece DX excelente (HMR instantâneo)
- Tailwind reduz tempo de estilização e garante consistência
- TypeScript elimina uma categoria inteira de bugs em runtime
- Supabase tem tier gratuito generoso para desenvolvimento e early stage

**Negativas / Trade-offs:**
- Dependência do Supabase como vendor — migração futura tem custo
- Lógica de negócio complexa precisa ir para Edge Functions (Deno) ou cliente — não há servidor Node próprio no início
- Supabase tem limites de conexão no tier gratuito (100 conexões)
- Edge Functions do Supabase rodam em Deno — diferente do Node.js padrão do time
- Tailwind pode gerar classes longas em componentes complexos — requer disciplina com componentes reutilizáveis
- SSR/SEO não é nativo — se SEO se tornar crítico, migração para Next.js seria necessária

**Mitigações:**
- Abstrair chamadas ao Supabase em módulos/serviços para facilitar migração futura
- Documentar em `memory/restrictions.md` os limites do tier atual do Supabase
- Usar Tailwind com disciplina: extrair classes repetidas em componentes (`@apply` ou componentes React)
