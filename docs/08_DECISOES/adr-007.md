# ADR-007: Theming e White-label por Estabelecimento

**Status:** Proposto (documentação — implementação não iniciada)
**Data:** 2026-07-06
**Decisores:** Matheus (founder)
**Supersede:** N/A (fecha o "ADR de theming/CSS pendente" citado nas decisões 017/018)
**Supersedido por:** N/A

## Contexto

A decisão 017 define o GastroMundi como SaaS white-label: cada estabelecimento pode ter identidade visual própria (marca, logo, cores, nome) por cima da plataforma. A decisão 018 pediu para **separar CSS do JSX** justamente para viabilizar isso, mas deixou o padrão concreto (CSS Modules? arquivos `.css` co-localizados? tokens por tenant?) para "um ADR de theming/CSS pendente" — este documento fecha essa pendência.

Hoje o design system inteiro é hardcoded em `src/constants/colors.js` (um objeto `C` com hex fixos, importado e usado como `style={{ color: C.accent }}` em centenas de lugares) e `src/constants/sizes.js` (tokens responsivos, não visuais). Não existe nenhum mecanismo de tema dinâmico. F018 já pede uma "revisão completa da estrutura de CSS" com base para theming — este ADR é o padrão que essa revisão deve seguir.

## Decisão

### 1. Tokens de tema como CSS Custom Properties, não mais objeto JS estático

`C` (`src/constants/colors.js`) deixa de ser a fonte final de valores de cor consumidos inline; passa a ser a lista dos **nomes de token** (`--gm-accent`, `--gm-bg`, `--gm-card`, etc.), com os valores atuais como default global:

```css
/* src/styles/tema.css — aplicado no <html> ou root da árvore, valores default (marca GastroMundi) */
:root {
  --gm-bg: #070b14;
  --gm-card: #0e1220;
  --gm-surface: #161b2c;
  --gm-border: #28324d;
  --gm-accent: #7c3aed;
  --gm-green: #10b981;
  --gm-red: #ef4444;
  --gm-blue: #3b82f6;
  --gm-text: #eef2f7;
  --gm-muted: #9aa8c4;
  /* ...restante dos tokens de colors.js */
}
```

Um tenant com branding próprio recebe um `<style>` inline pequeno (gerado a partir de `tenant.tema` vindo do bootstrap — ver seção 2) que sobrescreve só os tokens customizados, por cima do default:

```html
<style id="gm-tenant-theme">
  :root { --gm-accent: #0ea5e9; --gm-logo-url: url(...); }
</style>
```

Isso evita duplicar toda a folha de estilo por tenant — só as diferenças são enviadas.

### 2. Onde o tema do tenant é definido e como chega ao front

Novo campo em `public.tenants` (ADR-005): `tema jsonb NOT NULL DEFAULT '{}'` — guarda só os tokens que o tenant sobrescreve (ex.: `{"accent": "#0ea5e9", "nome_exibicao": "Pizzaria do João", "logo_url": "..."}`). Vem junto no retorno de `bootstrap_tenant()` (mesma RPC do ADR-005). No boot do app, antes de renderizar a árvore React, um pequeno efeito injeta esses tokens como CSS custom properties no `:root` (ou aplica a classe/atributo `data-tenant-theme` se preferirmos escopar por container em vez de global — ver trade-off abaixo).

Campos de branding cobertos nesta fase: cor de destaque (`accent`), nome de exibição, logo (URL), e favicon. Paleta completa customizável (todas as cores) fica para quando houver demanda real — a maioria dos clientes provavelmente só quer "minha cor" e "minha logo", não uma paleta inteira.

### 3. CSS separado do JSX (decisão 018) — padrão adotado: arquivo `.css` co-localizado

Entre CSS Modules, styled-components e arquivo `.css` puro co-localizado, a escolha é **arquivo `.css` puro, mesmo nome do componente, na mesma pasta** (`ClientesView.jsx` + `ClientesView.css`), com classes em `kebab-case` prefixadas pelo nome do componente (`.clientes-view__header`) para evitar colisão sem precisar de CSS Modules:

- **Por quê não CSS Modules:** exigiria mudar a config do Vite/loader e uma convenção de import (`import styles from "./X.module.css"`) em todo componente novo; o ganho (hashing automático de classes) não compensa a fricção extra num projeto que já não tem colisões de nome hoje (poucos componentes, nomes já descritivos).
- **Por quê não styled-components/Emotion:** nova dependência de runtime (custo de bundle, CLAUDE.md prioriza gratuito **e** enxuto), e o projeto já usa Tailwind como utilitário pontual — misturar três abordagens de estilo é pior que uma só.
- **Convenção de token:** todo valor de cor num `.css` novo referencia `var(--gm-*)`, nunca hex fixo — é isso que torna o componente adaptável por tenant automaticamente, sem precisar reescrever nada quando o tema mudar. `src/constants/colors.js` continua existindo, mas **desde o F018 não guarda mais hex**: `C.accent === "--gm-accent"` (nome do token), de modo que `var(${C.accent})` e `alfa(C.accent, "44")` emitem sempre a CSS var — o browser resolve por tenant. Não há mais dois lugares com o valor hex; a fonte única é `src/styles/tema.css` (invariante travado por `src/constants/colors.test.js`).
- Este é o mesmo padrão já aplicado nesta sessão em `DesempenhoReport.jsx` + `DesempenhoReport.css` (F011) — ADR formaliza o que já foi praticado.

### 4. Blend com transparência: `color-mix()` sobre `var(--gm-*)` (fecha a Leva 1 do F018)

O código legado usa hex + sufixo de alfa (`${C.accent}44`) pra transparência — truque que **não funciona** com CSS Custom Properties (`var(--gm-accent)44` não é uma cor válida). Padrão adotado: `color-mix()`, convertendo o sufixo hex pra porcentagem equivalente (arredondada ao inteiro mais próximo — ex. `44` ≈ 27%, `22` ≈ 13%, `1a` ≈ 10%):

```js
// src/constants/colorAlfa.js
alfa(C.accent, "44") // → "color-mix(in srgb, var(--gm-accent) 27%, transparent)"
```

- Se a cor de entrada bate com um token de marca conhecido (`C.xxx`), o blend usa `var(--gm-*)` e segue o tema do tenant.
- Se não bate (cor semântica fixa, ex. `AMBER` de alerta de tempo em `ComandaGrid`), usa a cor literal — não é branding, não deveria mudar por tenant mesmo.
- Arredondar o sufixo hex pra porcentagem inteira introduz uma diferença de opacidade de fração de ponto percentual (imperceptível visualmente) — aceito conscientemente em troca de não ter mais nenhuma cor presa fora do tema.
- Todo `.css`/`.jsx` novo que precisar de transparência usa `alfa()`, nunca volta ao truque de concatenação hex.

### 5. Escopo desta fase vs. F015/F018

- **F018 (revisão de CSS/responsivo)** é o trabalho de *migrar* as telas existentes (hoje 100% inline style) para o padrão acima — não faz parte deste ADR decidir a ordem/telas, só o padrão-alvo.
- **F015 (layouts de impressão)** — comprovantes, comandas de cozinha, notas fiscais — usam os mesmos tokens de tema (nome/logo do tenant no cabeçalho do comprovante), mas são HTML gerado à parte (`imprimirComanda` em `CheckoutView.jsx` já monta um documento HTML próprio para impressão) — o hook de branding aqui é simplesmente interpolar `tenant.tema.nome_exibicao`/`logo_url` no template ao montar o HTML, não CSS custom properties (janela de impressão não compartilha o `:root` da SPA).

## Alternativas Consideradas

- **CSS-in-JS (styled-components/Emotion) com tema via ThemeProvider:** rejeitado — dependência nova, runtime overhead, e o padrão de "objeto de cores + inline style" já é 90% do caminho para custom properties sem trocar de paradigma.
- **Tailwind com tema customizado via `tailwind.config` por build:** rejeitado — Tailwind gera CSS estático no build; teria que gerar **um build por tenant**, o que contradiz "um único codebase" (decisão 017). Custom properties resolvidas em runtime são a única forma de trocar tema sem rebuild.
- **Um tema JSON completo por tenant (toda paleta customizável):** adiado — mais flexível, mas mais superfície de suporte ("cliente escolheu cores ilegíveis") e não há demanda comprovada ainda; começar só com `accent`/nome/logo e expandir se pedirem.
- **Escopar tema por atributo (`[data-tenant="x"]`) em vez de `:root` global:** considerado para quando existir mais de um tenant renderizado na mesma sessão (ex.: um painel de administração da plataforma vendo vários tenants); não é o caso hoje (um tenant por sessão de app), mas é o padrão a adotar se isso mudar — registrado aqui para não esquecer.

## Consequências

**Positivas:**
- Branding por tenant vira "sobrescrever alguns custom properties", sem rebuild, sem fork de código — exatamente o que a decisão 017 pede.
- CSS separado do JSX (decisão 018) some junto com um objetivo concreto (theming), não fica sendo "boa prática" solta.
- Baixo custo: nenhuma dependência nova, só CSS puro + uma tabela.
- **F018 fechado: fonte única de verdade de cores.** `colors.js` deixou de ser uma segunda cópia hardcoded da paleta (guardava os mesmos hex que `tema.css`, sincronizados na mão) e passou a ser só o índice de nomes de token (`C.accent === "--gm-accent"`). Editar uma cor de marca agora é editar `tema.css`, um lugar só; o invariante é travado por teste (`colors.test.js`). Sobrescrever um `--gm-*` (ou `tenant.tema`) recolore tudo, inclusive o que vinha de `colors.js` (ícones via `currentColor`, blends via `color-mix`).

**Negativas / Trade-offs:**
- Toda tela nova precisa lembrar de usar `var(--gm-*)` em vez de hex fixo — sem enforcement automático (lint) nesta fase; considerar um lint customizado no futuro se o time crescer.
- Ícones (`react-icons`) recolorem por CSS nativo: `color={varColor(C.x)}` vira `style.color` no `<svg>` com `fill/stroke="currentColor"`. Casos que o CSS não alcança (canvas, lib de gráfico por prop, `<meta theme-color>`) — hoje inexistentes no código — devem usar o hook reativo `useCor("--gm-*")` (`src/lib/useCorHook.js`), nunca `colors.js` com hex.
- Restam alguns hex **literais semânticos/categóricos** em JS (AMBER `#f59e0b` de alertas de tempo/severidade do Jarvas, paletas categóricas de tipo de imposto e de método de pagamento, escala de força de senha, branco `#fff` de texto sobre cor) — **nunca estiveram em `colors.js`**, não são identidade de marca e, por decisão desta seção (§4), permanecem fixos (não recoloram por tenant). Não fazem parte da dupla-fonte que o F018 eliminou; se um dia virarem branqueáveis, promover a token (`--gm-warning` etc.).
- `logo_url`/`nome_exibicao` custom por tenant exigem upload/armazenamento de arquivo (Supabase Storage) — trabalho adicional não coberto por este ADR (fica para o plano técnico decidir se entra na primeira fase ou depois).
