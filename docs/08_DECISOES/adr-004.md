# ADR-004: Junção da fundação documental com o app em produção — stack real prevalece

**Status:** Aceito
**Data:** 2026-07-03
**Decisores:** Matheus (founder)
**Supersede:** [ADR-002](./adr-002.md) (parcialmente — arquitetura de acesso a dados e auth)
**Supersedido por:** N/A

## Contexto

Existiam dois projetos paralelos:

1. **Fundação (repo koraVReplit):** documentação completa (`memory/` + `docs/`), design system, regras de negócio dos módulos, modelagem multi-tenant e a decisão ADR-002 (API Express contract-first + Drizzle + Clerk). Código: apenas scaffold.
2. **App real (este repo, `gastromundi`):** PDV GastroMundi **em produção** — React 18 + Vite + **Supabase direto no frontend** (SDK, Auth, RLS, realtime), React Router v6, deploy na Vercel. Schema real em `supabase/schema.sql` + `supabase/migrations/` (users, products, pending, sales, fechamentos, logs, config, mesas...).

O ADR-002 havia descartado o "Caminho B" (frontend → Supabase direto), mas o app real foi construído e funciona exatamente nesse caminho. Manter a decisão contra a realidade de produção geraria documentação mentirosa.

## Decisão

- A fundação (`memory/` + `docs/`) passa a viver **neste repositório**, que é a fonte de verdade única do GastroMundi.
- **A stack real prevalece nesta fase:** React 18 + Vite + Supabase (SDK direto, Supabase Auth, RLS) + React Router v6 + Vercel. O ADR-002 fica **suspenso como arquitetura vigente** e é rebaixado a **direção de roadmap** (migração para API própria contract-first + Drizzle quando escala/equipe justificarem).
- **Schema real:** `supabase/schema.sql` + `supabase/migrations/` são a fonte de verdade do banco em produção. A modelagem multi-tenant de `docs/04_MODELAGEM/` (estabelecimentos/perfis/membros) e o schema Drizzle do repo da fundação ficam como **modelo-alvo** para a evolução multi-tenant.
- O app hoje é **single-tenant por instância**; a evolução para multi-tenant (decisão 002) segue no roadmap.
- Tudo que é independente de stack aplica-se imediatamente: identidade, regras de negócio dos módulos, design system, fluxos, backlog e a spec do **Jarvas** (IA transversal orientada a eventos — insight/alerta/sugestão, nunca executa sem confirmação).

## Alternativas Consideradas

- **Migrar o app para o monorepo (ADR-002 na íntegra):** descartado por ora — migração grande e arriscada de um app em produção, sem ganho imediato.
- **Manter dois repositórios:** descartado — duas fontes de verdade divergem inevitavelmente.

## Consequências

**Positivas:**
- Documentação e código no mesmo lugar, coerentes com a realidade de produção.
- O app ganha governança (memory/, ADRs) e o desenho de IA pronto para implementar o Jarvas.
- Roadmap arquitetural explícito em vez de decisão contraditória.

**Negativas / Trade-offs:**
- Lógica de negócio permanece no frontend + RLS até a eventual migração (menos camadas, mais acoplamento ao Supabase).
- `docs/01_ARQUITETURA/`, `docs/04_MODELAGEM/` e `docs/07_APIS/` descrevem parcialmente o modelo-alvo, não o estado atual — cada doc deve indicar isso até convergir.
