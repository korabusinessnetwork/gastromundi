# ADR-006: Billing e Enforcement de Assinatura

**Status:** Aceito — questões em aberto resolvidas pelo founder em 2026-07-06 (ver seção "Questões em Aberto")
**Data:** 2026-07-06
**Decisores:** Matheus (founder)
**Supersede:** N/A
**Supersedido por:** N/A

> **Nota de escopo (resolvida):** a mensagem original que gerou este ADR foi cortada em "Custo: construir toda a lógica agora com [...]". O founder confirmou a leitura assumida: **construir toda a lógica de vigência/carência/bloqueio agora** (é só Postgres + regra de negócio, custo zero, cabe no free tier do Supabase) **sem integrar gateway de pagamento pago ainda** — a confirmação de pagamento entra manualmente nesta fase (ver Questão 4).

## Contexto

F016 pede um mecanismo de **enforcement de assinatura**: cada tenant paga um ciclo (ex.: 30 dias); perto do vencimento, um aviso; ao vencer sem renovar, o sistema **bloqueia** com a mensagem "Sua mensalidade está atrasada". O texto do pedido é explícito: *"a checagem de vigência mora no backend (RLS/RPC), nunca só no front"* — ou seja, esconder a UI não basta; a Doc de fluxo já existente (`docs/05_FLUXOS/billing-flow.md`) descreve esse fluxo em termos genéricos (planos free/pro/enterprise, e-mail, gateway) que precisam ser atualizados para bater com a arquitetura real (ADR-005) e com a realidade de custo do projeto (sem gateway pago ainda, sem e-mail transacional configurado).

Este ADR levantou três questões em aberto (bloqueio total vs. parcial, dias de carência, mecanismo de agendamento) além da leitura da mensagem cortada sobre custo. **Todas foram resolvidas pelo founder em 2026-07-06** — ver seção "Questões em Aberto" para as respostas definitivas, incorporadas na Decisão abaixo.

## Decisão

### 1. Modelo de dados: assinatura ≠ add-on (mensalidade separada de cobrança avulsa)

```sql
CREATE TABLE public.assinaturas (
  tenant_id        uuid PRIMARY KEY REFERENCES public.tenants(id),
  plano_codigo     text NOT NULL REFERENCES public.planos(codigo),
  valor_mensal     numeric NOT NULL,
  ciclo_dias       integer NOT NULL DEFAULT 30,
  data_inicio      date NOT NULL,
  data_vencimento  date NOT NULL,          -- próxima data em que o ciclo pago expira
  status           text NOT NULL DEFAULT 'ativo'
                     CHECK (status IN ('ativo', 'carencia', 'bloqueado', 'cancelado')),
                     -- Cache/observabilidade (listagens admin), NÃO é a fonte de verdade do
                     -- enforcement — ver seção 3: o status real é sempre derivado na consulta
                     -- a partir de data_vencimento + carencia_dias, exceto 'cancelado' (manual).
  carencia_dias    integer NOT NULL DEFAULT 3,  -- decidido pelo founder (Questão 2) — default global, pode variar por tenant no futuro
  ultima_renovacao date,
  criado_em        timestamptz NOT NULL DEFAULT now()
);

-- Histórico de ciclos pagos — auditoria e futura integração com gateway
CREATE TABLE public.assinaturas_pagamentos (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     uuid NOT NULL REFERENCES public.tenants(id),
  competencia   date NOT NULL,        -- ciclo a que se refere (ex.: início do período pago)
  valor         numeric NOT NULL,
  metodo        text,                 -- 'pix_manual' | 'boleto_manual' | 'gateway' (futuro)
  confirmado_por text,                -- quem deu baixa manual (admin da plataforma)
  confirmado_em timestamptz NOT NULL DEFAULT now()
);
```

Add-ons pagos (NF-e/TEF, decisão 019 e ADR-005) têm cobrança e ciclo **independentes** — não vivem em `assinaturas`. Cada add-on ativo pode, no futuro, ter seu próprio valor e ciclo em `tenant_addons` (ADR-005) sem afetar a data de vencimento da mensalidade do plano. Isso evita o cenário ruim de "cliente atrasou o TEF e perdeu acesso ao PDV inteiro" — inadimplência de add-on bloqueia **só o add-on**, inadimplência do plano bloqueia **o sistema**.

### 2. Ciclo de vida da assinatura

```
ativo ──(vencimento passa sem renovação)──> carencia ──(carência esgota sem renovação)──> bloqueado
  ↑                                              │                                             │
  └──────────────── renovação confirmada ────────┴─────────────────────────────────────────────┘
```

- **Aviso pré-vencimento:** quando `hoje >= data_vencimento - 5 dias` (configurável), o bootstrap retorna um aviso (`assinatura.aviso: "vence_em_3_dias"`); a UI mostra um banner não bloqueante.
- **Carência:** no dia seguinte ao vencimento, o status **efetivo** (calculado, ver seção 3) passa a `carencia` — sistema continua funcionando normalmente, mas o banner vira um aviso mais forte ("mensalidade vencida, X dias para regularizar").
- **Bloqueio:** esgotados os `carencia_dias` (3 dias, Questão 2) sem renovação, o status efetivo vira `bloqueado` — dispara o enforcement da seção 3, **imediatamente**, mesmo sem nenhum job ter rodado (ver seção 3 — o cálculo não depende de job).
- **Renovação (nesta fase, manual — Questão 4):** o dono/admin da plataforma confirma o pagamento (Pix, transferência, etc. fora do sistema) e uma tela administrativa chama uma RPC `confirmar_renovacao_assinatura(tenant_id, competencia, valor, metodo)` que grava em `assinaturas_pagamentos`, atualiza `data_vencimento += ciclo_dias` e volta a coluna `status` (cache) para `ativo`. O ponto de extensão para gateway de pagamento fica pronto para, no futuro, chamar essa mesma RPC via webhook em vez de confirmação manual.

### 3. Enforcement — status derivado na consulta, nunca dependente de job (Questão 3, resolvida)

Duas camadas, a segunda é a que importa de verdade (o pedido foi explícito: *"nunca só no front"*):

**Camada UI (cortesia, não segurança):** o bootstrap retorna `assinatura.status_efetivo`; se `bloqueado`, o app renderiza uma tela cheia "Sua mensalidade está atrasada" no lugar de qualquer rota (`PrivateRoute` intercepta antes de montar `DesktopLayout` — inclusive antes do login concluir, ver seção 4). Rápido de construir, boa UX, mas **não impede** alguém de chamar a API do Supabase diretamente.

**Camada backend (fonte de verdade — cálculo em tempo real):** a correção do enforcement **não pode depender de um job ter rodado** (`pg_cron` pago/indisponível, ou um bootstrap que nunca aconteceu naquele dia não podem ser a única coisa impedindo uma venda inadimplente). Por isso:

- Uma função pura `public.calcular_status_assinatura(p_data_vencimento date, p_carencia_dias integer, p_hoje date DEFAULT current_date) RETURNS text` calcula o status **na hora**, sem ler nenhuma coluna de cache:
  ```sql
  CASE
    WHEN p_hoje <= p_data_vencimento                          THEN 'ativo'
    WHEN p_hoje <= p_data_vencimento + p_carencia_dias         THEN 'carencia'
    ELSE 'bloqueado'
  END
  ```
- `public.assinatura_ativa(tenant_id uuid) RETURNS boolean` (`SECURITY DEFINER`, `STABLE`) busca `data_vencimento`/`carencia_dias` de `assinaturas`, chama `calcular_status_assinatura(...)` e retorna `true` só se o resultado for `'ativo'` ou `'carencia'` — **exceto** se a coluna `status` (cache) já estiver `'cancelado'` manualmente, que sempre bloqueia independente de data. Essa função entra como cláusula adicional (`AND`) nas políticas `WITH CHECK`/`USING` de **toda** operação nas tabelas operacionais (ver seção 4 — bloqueio total, não só escrita), o mesmo padrão de composição de política já usado no ADR-005 para gating por módulo.
- A coluna `assinaturas.status` (armazenada) é **só um cache** para listagens administrativas e para acionar avisos — ela é atualizada por uma checagem **lazy no bootstrap** (Questão 3, resolvida: sem `pg_cron` nesta fase): a cada `bootstrap_tenant()`, se `calcular_status_assinatura(...)` difere do `status` armazenado, o bootstrap chama `sincronizar_status_assinatura(tenant_id)` para atualizar a coluna. Mas **o enforcement em si nunca lê essa coluna para decidir bloquear** — sempre recalcula a partir das datas. Isso significa: mesmo que ninguém abra o app por dias e o cache fique desatualizado, a primeira tentativa de escrita depois do vencimento+carência já falha corretamente.

### 4. O que continua acessível durante o bloqueio (Questão 1, resolvida: bloqueio TOTAL)

**Decisão do founder:** bloqueio é **total** — `bloqueado` impede tudo, inclusive login/leitura e o fechamento de um caixa já aberto. Não há exceção operacional. A recomendação técnica original (permitir leitura e fechamento de caixa) foi avaliada e conscientemente descartada pelo founder — ver trade-off registrado em Consequências.

## Alternativas Consideradas

- **Checagem só no front-end (esconder rotas quando vencido):** rejeitada explicitamente pelo pedido original e pelas restrições de segurança do projeto (CLAUDE.md) — qualquer chamada direta à API do Supabase ignoraria o bloqueio.
- **Integrar gateway de pagamento (Stripe/Mercado Pago/PagSeguro) já nesta fase:** adiado por custo (Restrições de Custo) — o volume de clientes hoje não justifica a integração e o tempo de implementação; a confirmação manual cobre o caso de uso enquanto a base de clientes é pequena. Fica registrado como upgrade natural: trocar "confirmação manual" por "webhook do gateway chamando a mesma RPC" não exige mudar o modelo de dados.
- **Um único status por tenant cobrindo plano + add-ons (bloqueio único):** rejeitada — inadimplência de um add-on avulso (ex.: não pagou o TEF do mês) não deveria travar o PDV inteiro; só a mensalidade do plano trava o sistema.
- **Sem período de carência (bloqueio imediato no vencimento):** rejeitada — o pedido original já especifica carência explicitamente; bloqueio imediato também é mais hostil ao cliente do que o padrão de mercado (a maioria dos SaaS dá alguns dias).
- **Status de enforcement armazenado em coluna, atualizado só por job (`pg_cron`/cron externo):** rejeitada — faria a segurança depender de um job ter rodado; se o job falhar ou atrasar, um tenant inadimplente continuaria operando. Substituída pelo cálculo em tempo real (seção 3): o job/lazy check só mantém um cache para UI/relatórios, nunca é a fonte da decisão de bloquear.
- **Bloqueio parcial (permitir leitura/fechamento de caixa mesmo bloqueado):** era a recomendação técnica original; **rejeitada pelo founder** em favor do bloqueio total, por simplicidade de implementação e por tratar inadimplência como interrupção completa e inequívoca (sem zona cinzenta operacional).

## Consequências

**Positivas:**
- Enforcement real (Postgres) e **correto mesmo sem job algum ter rodado** — o cálculo de status é sempre feito na consulta, a partir de `data_vencimento`/`carencia_dias`, nunca de um cache que poderia estar desatualizado.
- Mensalidade do plano e add-ons avulsos falham de forma independente — inadimplência de um não trava o outro.
- Sem dependência de `pg_cron` nesta fase — a checagem lazy no bootstrap (Questão 3) roda 100% dentro do Supabase free tier, sem agendador externo.
- Caminho de upgrade para gateway de pagamento real não exige redesenhar o modelo — só trocar quem chama `confirmar_renovacao_assinatura`.
- Bloqueio total é a política mais simples de implementar (uma cláusula `AND` cobrindo todas as operações, sem exceção por tabela/ação) e sem ambiguidade para o usuário final.

**Negativas / Trade-offs:**
- Confirmação manual de pagamento não escala para muitos clientes — é uma decisão consciente de adiar custo, com prazo de revisão implícito (quando o número de tenants crescer, reavaliar o gateway).
- Bloqueio total significa que um caixa aberto no momento do bloqueio fica sem poder ser fechado pelo app até a renovação — risco operacional aceito explicitamente pelo founder (Questão 1) em troca de simplicidade e clareza.
- A checagem lazy no bootstrap só atualiza o cache de status quando alguém abre o app — entre uma abertura e outra, o cache pode ficar visualmente desatualizado (ex.: um admin olhando uma listagem), mas isso **não afeta a segurança** (seção 3), só a precisão de relatórios administrativos até o próximo bootstrap.

## Questões em Aberto — RESOLVIDAS pelo founder em 2026-07-06

1. **Bloqueio total vs. parcial → TOTAL.** `bloqueado` impede tudo, inclusive login/leitura e fechar um caixa já aberto. Sem exceção operacional nesta fase.
2. **Carência padrão → 3 dias.** `assinaturas.carencia_dias DEFAULT 3`.
3. **Mecanismo de agendamento → checagem LAZY no bootstrap, sem `pg_cron`.** E, mais importante: o enforcement em si (`assinatura_ativa`) **nunca depende do job ter rodado** — ele recalcula o status a partir de `data_vencimento`/`carencia_dias` a cada chamada (seção 3). O job/lazy check serve só para manter a coluna `status` (cache) e os avisos em dia, não para decidir bloquear.
4. **Custo/gateway → confirmado.** Lógica completa (vigência/carência/bloqueio) construída agora, dentro do Supabase free tier; gateway de pagamento pago fica adiado; renovação é confirmada manualmente por um admin da plataforma nesta fase.

**Ainda em aberto (não bloqueia a Fase 1 do plano técnico, só fases posteriores):**
- Notificação pré-vencimento por e-mail/SMS — hoje o projeto não tem provedor configurado (custo, ver Restrições); banner dentro do app é suficiente por ora.
