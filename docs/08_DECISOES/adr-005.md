# ADR-005: Planos, Gating por Módulo e Add-ons Pagos

**Status:** Proposto (documentação — implementação não iniciada)
**Data:** 2026-07-06
**Decisores:** Matheus (founder)
**Supersede:** [Decisão 008](../../memory/decisions.md) (parcialmente — nomes e granularidade dos planos)
**Supersedido por:** N/A

## Contexto

O GastroMundi é, por decisão de produto (decisão 017), um **SaaS multi-estabelecimento white-label**: um único codebase vendido para vários estabelecimentos, cada um com seu plano contratado. Hoje isso não existe — o app é single-tenant (ADR-004) e todo o gating de acesso é feito só por **papel** (`src/constants/roles.js`, `ROLES.<papel>.permissions`), sem nenhum conceito de plano.

O backlog (F013) pede planos com cinco níveis (Básico, Simples, Médio, Alto, Avançado), cada um superconjunto do anterior, e a decisão 019 já estabeleceu que **NF-e e TEF são add-ons pagos transversais** — disponíveis em qualquer plano mediante contratação à parte, não amarrados a um tier. Precisamos de uma arquitetura de gating que:

- Tenha **uma única fonte de verdade** para "o que cada plano/add-on libera" — nada de `if (plano === 'alto')` espalhado pelo código.
- Bloqueie de verdade no backend (RLS/RPC) — esconder um item no Sidebar não impede alguém de chamar a API diretamente.
- Componha corretamente **plano** (bundle por assinatura) com **add-ons** (funcionalidades avulsas, pagas à parte, cruzando todos os planos).
- Não force, só para isso, a migração completa para multi-tenant (decisão 002) — que é maior, mais arriscada, e não é pré-requisito estrito para vender planos a um único estabelecimento por vez.

## Decisão

### 1. Nomenclatura e granularidade dos planos (revisa decisão 008)

A decisão 008 registrou os planos como `free / pro / enterprise`. Este ADR **substitui essa nomenclatura** pelos cinco tiers já usados no backlog, cada um **superconjunto estrito** do anterior:

| Plano | Módulos incluídos (além do anterior) |
|-------|----------------------------------------|
| **Básico** (piso) | Cardápio/Produtos, PDV, Caixa |
| **Simples** | + Estoque, Pedidos |
| **Médio** | + Mesas/Comandas, Cozinha (KDS), Alertas (estoque mínimo, etc.) |
| **Alto** | + Financeiro, Clientes/Fiado, Relatórios |
| **Avançado** | + Jarvas, Multi-loja, Fiscal/integrações nativas (não confundir com os add-ons da seção 3) |

Um estabelecimento no plano Alto tem automaticamente tudo do Básico, Simples e Médio — não existe "módulo do Médio que falta no Alto". Essa regra de superconjunto é o que permite ao registro central ser uma simples tabela de associação, sem lógica condicional.

### 2. Registro central plano → módulos (fonte única)

Duas tabelas novas no Postgres, seed via migration, **somente leitura pelo app**:

```sql
CREATE TABLE public.planos (
  codigo      text PRIMARY KEY,        -- 'basico' | 'simples' | 'medio' | 'alto' | 'avancado'
  nome        text NOT NULL,           -- rótulo amigável ("Básico", "Simples"...)
  ordem       integer NOT NULL,        -- 1..5, define o "superconjunto" (ordem >= ordem_requerida)
  ativo       boolean NOT NULL DEFAULT true
);

CREATE TABLE public.planos_modulos (
  plano_codigo   text NOT NULL REFERENCES public.planos(codigo),
  modulo_codigo  text NOT NULL,        -- 'cardapio' | 'pdv' | 'caixa' | 'estoque' | 'pedidos' |
                                        -- 'mesas_comandas' | 'cozinha' | 'alertas' | 'financeiro' |
                                        -- 'clientes' | 'relatorios' | 'jarvas' | 'multiloja' | 'fiscal_integracoes'
  PRIMARY KEY (plano_codigo, modulo_codigo)
);
```

`planos_modulos` é povoada uma vez, respeitando o superconjunto (ex.: a linha do "alto" repete todos os módulos do "médio" mais os quatro novos). Essa redundância é proposital: uma junção simples (`plano_codigo = tenant.plano_codigo`) resolve "quais módulos esse tenant tem", sem precisar calcular superconjuntos em tempo de execução.

### 3. Add-ons pagos (decisão 019) como eixo ortogonal

Add-ons **não vivem em `planos_modulos`** — eles são flags por tenant, independentes de plano:

```sql
CREATE TABLE public.addons (
  codigo  text PRIMARY KEY,     -- 'nfe' | 'tef'
  nome    text NOT NULL,
  modulos text[] NOT NULL       -- módulos que o add-on libera quando ativo (ex.: 'fiscal_emissao', 'tef_pagamento')
);

CREATE TABLE public.tenant_addons (
  tenant_id     uuid NOT NULL REFERENCES public.tenants(id),
  addon_codigo  text NOT NULL REFERENCES public.addons(codigo),
  ativo         boolean NOT NULL DEFAULT true,
  ativado_em    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (tenant_id, addon_codigo)
);
```

Os módulos efetivamente liberados para um tenant são a **união** dos módulos do seu plano com os módulos dos add-ons ativos:

```
modulos_do_tenant(t) = modulos_do_plano(t.plano_codigo) ∪ modulos_dos_addons_ativos(t)
```

Isso resolve de forma limpa a aparente tensão entre "Avançado inclui fiscal/integrações" (decisão deste ADR) e "NF-e/TEF são add-ons em todos os planos" (decisão 019): são coisas diferentes. `fiscal_integracoes` no plano Avançado cobre recursos nativos de escala (ex.: consolidação fiscal multi-CNPJ, integrações de delivery/marketplace); os add-ons `nfe`/`tef` continuam comprável à parte em **qualquer** plano, inclusive Básico. Nenhum código pergunta "seu plano é X" para liberar NF-e/TEF — pergunta apenas "seu tenant tem o add-on ativo".

### 4. Bootstrap do tenant

Uma única RPC, `bootstrap_tenant()` (`SECURITY DEFINER`, `STABLE`), chamada uma vez no login/carregamento do app (dentro da já existente `bootstrap()` de `AppContext.jsx`), retorna:

```jsonc
{
  "tenant_id": "...",
  "plano_codigo": "alto",
  "addons_ativos": ["nfe"],
  "modulos_disponiveis": ["cardapio", "pdv", "caixa", "estoque", "pedidos",
                          "mesas_comandas", "cozinha", "alertas",
                          "financeiro", "clientes", "relatorios",
                          "fiscal_emissao"],
  "assinatura": { "status": "ativo", "vencimento": "2026-08-05", "carencia_ate": null }
}
```

O front-end guarda isso num novo slice do `AppContext` (`tenant`) e nunca recalcula a união plano+add-ons no cliente — só consome a lista `modulos_disponiveis` já pronta. `assinatura` é o gancho para o ADR-006 (billing/enforcement).

### 5. Gating em três camadas

| Camada | Onde | O que faz | Nível de confiança |
|--------|------|-----------|---------------------|
| **1. UI (Sidebar/rotas)** | `Sidebar.jsx`, `routes/index.jsx`, `PrivateRoute` | Esconde itens de menu e bloqueia rotas cujo módulo não está em `tenant.modulos_disponiveis` | UX apenas — não é segurança |
| **2. RLS/RPC (Postgres)** | Políticas `WITH CHECK`/`USING` das tabelas por módulo + função `public.tenant_tem_modulo(tenant_id, modulo_codigo)` | Bloqueia de verdade: mesmo uma chamada direta à API REST do Supabase falha se o tenant não tem o módulo | Fonte de verdade de segurança |
| **3. Registro central** | `planos_modulos` + `tenant_addons` | É a única definição de "quem tem o quê"; as camadas 1 e 2 só leem daqui | Fonte única (evita duplicidade/drift) |

A camada 1 continua combinando com o gating por **papel** já existente (`ROLES.<papel>.permissions`) — um item só aparece se **o papel do usuário permite** *e* **o plano/add-on do tenant inclui o módulo**. São dois filtros independentes que se multiplicam (E lógico), não um substituindo o outro.

A camada 2 é aditiva às políticas de RLS já existentes (papel + tenant ownership, decisão 002): a checagem de plano entra como mais uma cláusula `AND` nas políticas de escrita das tabelas por módulo (ex.: `lancamentos`, `clientes`, `estoque`), via a função `tenant_tem_modulo(...)`.

### 6. Pré-requisito mínimo: tabela `tenants` (sem migração multi-tenant completa)

Hoje não existe nenhuma tabela `tenants` nem coluna `tenant_id` em lugar nenhum do schema real (ADR-004: app é single-tenant). Fazer join com `planos`/`addons` exige *algum* jeito de identificar "o tenant atual". Este ADR propõe o menor passo possível, sem forçar a migração multi-tenant completa (que continua sendo trabalho futuro da decisão 002):

- Criar `public.tenants (id uuid, nome text, plano_codigo text, created_at timestamptz)` com **exatamente uma linha hoje** (a instalação atual do GastroMundi).
- **Não** adicionar `tenant_id` em todas as tabelas existentes agora — isso é a migração multi-tenant completa, fora de escopo deste ADR.
- `bootstrap_tenant()` resolve "o tenant atual" pela única linha existente (equivalente a `SELECT * FROM tenants LIMIT 1`), com o ponto de extensão já pronto para, no futuro, resolver por `auth.uid() → membership → tenant_id` quando o segundo cliente precisar da mesma instalação (multi-tenant real).
- Isso evita dois riscos: (a) fazer planos "de mentirinha" que não sobrevivem à migração multi-tenant real, e (b) atrasar F013 esperando uma migração muito maior que não é hoje o gargalo comercial.

## Alternativas Consideradas

- **`if (tenant.plano === 'alto')` espalhado nos componentes:** rejeitado — é exatamente o anti-padrão que o pedido original queria evitar; qualquer novo módulo exigiria caçar e editar N pontos do código, alto risco de esquecimento e de furo de segurança (checagem só no front).
- **Guardar o mapa plano→módulos como JSONB em `public.config`** (padrão já usado no app para `meios_pagamento`, `fichas_tecnicas` etc.): considerado por consistência com o resto do app, mas rejeitado para este caso específico porque o RLS precisa **consultar isso a cada escrita** (via função SQL) — uma tabela relacional simples é mais barata e mais fácil de indexar/auditar que abrir e parsear um blob JSONB toda vez. `config` continua sendo o lugar certo para configurações operacionais do dia a dia (não para o registro comercial de planos).
- **Multi-tenant completo agora, como pré-requisito de F013:** rejeitado nesta fase — maior, mais arriscado, adia a monetização sem necessidade; o app atende um estabelecimento por instância hoje, então uma tabela `tenants` de uma linha já desbloqueia o gating por plano sem esse risco.
- **Add-ons dentro de `planos_modulos` (como se fossem "módulos de um plano invisível"):** rejeitado — misturaria dois conceitos de cobrança diferentes (assinatura recorrente por tier vs. contratação avulsa) na mesma tabela, dificultando relatórios de receita separados por linha de produto.

## Consequências

**Positivas:**
- Um único lugar (`planos_modulos` + `tenant_addons`) define o que cada instalação pode usar — consistente com "registro central, fonte única" pedido no escopo.
- Gating real (RLS/RPC) impede bypass via chamada direta à API, não só escondendo botões.
- Add-ons e planos evoluem independentemente (lançar um add-on novo não mexe na tabela de planos, e vice-versa).
- Caminho de migração para multi-tenant real fica mais barato (a tabela `tenants` já existe; só falta popular com mais linhas e ligar `tenant_id` nas tabelas operacionais).

**Negativas / Trade-offs:**
- Duas tabelas novas de baixíssimo volume, mas que exigem migração e seed cuidadoso sempre que um módulo novo nascer (precisa decidir em qual plano ele entra).
- Enquanto não existir multi-tenant real, "tenant" é uma abstração de uma linha só — todo o desenho tem que ser validado de novo quando o segundo cliente aparecer (risco aceito e registrado aqui).
- Duas checagens (papel E plano) tornam o debug de "por que não vejo esse menu" um pouco menos óbvio — mitigado exibindo, na UI, qual dos dois filtros bloqueou (ver ADR de UX no plano técnico).
