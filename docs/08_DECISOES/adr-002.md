# ADR-002: Revisão da Stack — API Express (contract-first) + Drizzle + Clerk; Supabase como Postgres gerenciado

**Status:** Aceito
**Data:** 2026-06-16
**Decisores:** A preencher (founders + tech lead)
**Supersede:** [ADR-001](./adr-001.md) (parcialmente — ver "Decisão")
**Supersedido por:** N/A

## Contexto

O ADR-001 definiu a stack inicial como React + Vite + Tailwind + **Supabase** (BaaS), com o frontend acessando o Supabase diretamente via SDK e usando **Supabase Auth** + RLS. Ao iniciar a implementação no ambiente real (monorepo pnpm da Replit), surgiram dois fatos que exigem revisão:

1. O ambiente já entrega um **scaffold nativo**: API em **Express 5**, **Drizzle ORM**, **PostgreSQL** e um fluxo **contract-first** (OpenAPI → Orval → hooks React Query + schemas Zod). Descartá-lo para usar o SDK do Supabase direto significaria jogar fora infraestrutura pronta e a tipagem ponta a ponta.
2. A autenticação escolhida pelo time passou a ser **Clerk** (gerenciado pela Replit), e **não** o Supabase Auth. Amarrar a RLS do Supabase a tokens do Clerk exigiria uma ponte JWT de terceiros — frágil e custosa.

## Decisão

Adotamos o **Caminho A**: manter o scaffold nativo do monorepo e usar o Supabase apenas como **PostgreSQL gerenciado**.

| Camada | Tecnologia | Observação |
|--------|-----------|-----------|
| Frontend framework | React | **19.x** (corrige "18.x" do ADR-001) |
| Build tool | Vite | **7.x** (corrige "5.x") |
| Estilização | Tailwind CSS | **4.x** (corrige "3.x") |
| Roteamento | **wouter** | substitui React Router (já no catálogo do monorepo) |
| Estado global de UI | Context API | mantém decisão 003 |
| Estado de servidor | TanStack Query | hooks gerados pelo Orval |
| API / Backend | **Express 5** (`artifacts/api-server`) | contract-first via OpenAPI |
| Acesso a dados | **Drizzle ORM** | camada de serviços isola o acesso (revisa decisão 007) |
| Banco de dados | **PostgreSQL hospedado no Supabase** | conexão via `DATABASE_URL` |
| Autenticação | **Clerk** (gerenciado pela Replit) | cookie de sessão + middleware Express; substitui Supabase Auth |
| Validação de contratos | Zod | mantém decisão 006 |

### O que muda em relação ao ADR-001
- **Supabase deixa de ser BaaS** e passa a ser somente o host do Postgres. Não usamos Supabase Auth, Storage ou SDK no frontend nesta fase (reavaliáveis por novo ADR se necessário).
- **O frontend não fala direto com o banco**: toda leitura/escrita passa pela API Express (contract-first), que aplica validação (Zod) e isolamento por tenant.
- **Autenticação via Clerk** no lugar do Supabase Auth.
- **Roteamento via wouter** no lugar do React Router; versões de React/Vite/Tailwind corrigidas para as reais.

### O que se mantém do ADR-001
- React + Vite + Tailwind + TypeScript + PostgreSQL + RLS por `tenant_id`.
- Princípios: separação de responsabilidades, contrato explícito (Zod), baixo acoplamento (Event Bus), segurança por padrão.

## Multi-tenant e segurança
- Todas as tabelas com dados de estabelecimento carregam `tenant_id`.
- Isolamento aplicado em **duas camadas**: (1) na API/camada de serviços (o tenant ativo vem do vínculo do usuário Clerk em `membros`) e (2) opcionalmente em **RLS no PostgreSQL** via parâmetro de sessão (`SET app.tenant_id`).
- A `service_role`/credenciais administrativas nunca são expostas no frontend.

## Alternativas Consideradas
- **Caminho B — Frontend direto no Supabase SDK + RLS no Supabase + ponte Clerk→Supabase (JWT).** Mais fiel ao ADR-001 original, porém mais frágil (depende de JWT de terceiros), descarta o scaffold contract-first e duplica regras de negócio no cliente. **Rejeitado.**
- **Replit PostgreSQL nativo no lugar do Supabase.** Simples, mas o time optou explicitamente por Supabase. **Não adotado nesta fase.**

## Consequências
**Positivas:**
- Aproveita o scaffold do monorepo (Express + Drizzle + Orval + Zod) e a tipagem ponta a ponta.
- Auth robusta e de baixa fricção (Clerk gerenciado), sem ponte JWT.
- Regras de negócio centralizadas na API (não no cliente), facilitando auditoria e o Event Bus.

**Negativas / Trade-offs:**
- Mantemos um servidor Node próprio (mais que o modelo "só BaaS") — porém já provido pelo ambiente.
- Recursos nativos do Supabase (Realtime, Storage, Edge Functions) não são usados por ora; se necessários, exigem novo ADR.
